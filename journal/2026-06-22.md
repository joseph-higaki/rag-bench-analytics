## What got done
- Stood up the local Docker stack (Postgres 16, MinIO) for the first time.
- Ran `make seed` (163 files into MinIO) and `make ingest` (81 runs, 3,461 records, 58 questions into `raw` schema).
- Fixed two dbt bugs blocking `make dbt`:
  1. **Source schema not resolving:** `_sources.yml` used `var('raw_schema')` which resolved to the literal string `{{ env_var('RAW_SCHEMA', 'raw') }}` — `var()` returns strings, it doesn't re-render nested Jinja. Fixed by calling `env_var()` directly in `_sources.yml` and removing the unused `vars.raw_schema` from `dbt_project.yml`.
  2. **dim_corpus FK orphans:** 1,303 fact rows from pre-June 14 runs had no `corpus_build_id` in their manifest (the benchmark didn't emit it yet). `dim_corpus` filtered `WHERE corpus_build_id IS NOT NULL`, so no dim row existed for the null hash. Fixed by removing the filter (allowing an "unknown corpus" row) and relaxing the `not_null` test on `corpus_build_id`.
- Confirmed `make dbt` passes 70/70: seeds, staging views, mart tables, all data tests including FK relationships.
- Fixed stale venv shebang (pointed at old `biomedical-rag-bench-analytics` path).

## Decisions (and why)
- **`env_var()` directly in `_sources.yml`** rather than fixing the `var()` indirection — the indirection added no value (no other consumers of `raw_schema`) and the double-render trap is easy to hit again.
- **Unknown corpus row with null `corpus_build_id`** — standard dim modeling: fact FKs must always resolve. Alternative was coalescing to a sentinel string in the fact, but null is more honest (the data genuinely doesn't have it).
- **`make export` dropped from `make pipeline`** — per ADR-001, the dashboard reads marts via direct Postgres connection. Export is an optional fallback, not the main path. (Makefile target kept but not in the pipeline chain — to be updated.)
